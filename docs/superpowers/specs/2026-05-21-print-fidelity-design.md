# Print + Edit-Mode Fidelity — Design Spec

**Date:** 2026-05-21
**Author:** Codex (with jlgarcia@esoftsat.com)
**Status:** Approved for plan writing
**Scope:** Two related render-fidelity fixes shipped as one body of work.

---

## Motivation

Two fidelity problems make the editor feel rough around the edges:

1. **PDF pagination.** The print route reuses the email HTML, which has no concept of paged output. When the email content overflows one printed page, the browser breaks pages mid-content. The footer — being just another element in document flow — ends up wherever flow ended, often mid-page on the last page rather than pinned to the bottom. Product sections can also split across pages.

2. **Edit-mode visual divergence.** Inline edit affordances inflate or shift content while in edit mode. The bullet drag-grip occupies ~28px before each bullet's text, the CTA URL pencil icon makes buttons wider, and various hover controls push surrounding layout. The user sees one rendering while editing and a different one in preview / on export — undermining the WYSIWYG promise.

The two problems share an underlying principle: rendered output should match design intent regardless of viewing context. Bundling them keeps the diff focused on "render fidelity" and lets one round of manual smoke verify both.

---

## Goals

- The PDF generated by the print route has a header at the top margin of every page and a footer at the bottom margin of every page (including the last, even when the last page is sparse).
- Product sections and all other middle blocks never split across pages.
- The editor canvas at the same zoom level produces visually identical bullets, CTA buttons, and block widths in edit mode and preview mode.
- All five existing templates (Blank, GlobalTT, Newsletter, Announcement, Event Invite) print correctly.

## Non-goals (this spec)

- **Balanced page distribution.** Deferred until we see PagedJS output and decide whether sparse-last-page is acceptable.
- **Server-side PDF generation** (Puppeteer/Playwright). Client-side PagedJS produces sufficient fidelity.
- **Block palette UI / drop zones.** Phase 3 concern.
- **Page-size configurability** (A4 vs Letter). A4 stays the default.
- **PDF metadata** (outline, anchors, document title). Future "Pro PDF" feature.
- **PagedJS via CDN.** Vendoring is the chosen approach.
- **Rename `section-wrap` → `block-wrap` globally.** Optional cleanup, deferred.

## Architecture overview

### Print pipeline

```
ProjectData
  │
  ├──► renderEmail()       ─► email-safe HTML (Outlook/Gmail constraints)
  │
  └──► renderPrintDocument() ─► print-optimized HTML
                                  │
                                  ▼
                            buildPrintHtml()  (wraps with PagedJS + toolbar)
                                  │
                                  ▼
                            /w/[slug]/p/[id]/print route response
                                  │
                                  ▼
                            browser loads → PagedJS paginates → user prints
```

`renderEmail` and `renderPrintDocument` share the source-of-truth `ProjectData` but produce different HTML. Email HTML uses table-based MSO-safe markup; print HTML uses modern CSS (CSS Paged Media via PagedJS polyfill).

### Edit-fidelity pipeline

```
ProjectData
  │
  └──► <PreviewBody>
         │
         ├──► <ProductSectionView>, <HeroBlockView>, <ArticleView>, ...
                │
                ├──► <EditableText>     ─┐
                ├──► <EditableLink>     ─┤  edit-only affordances
                ├──► <EditableImage>    ─┤  must NOT occupy normal flow
                └──► <EditableBulletList>─┘
```

The fix is mechanical: every edit-only affordance becomes absolutely positioned (or `display: none` until hover/focus), revealed via `.section-wrap:hover` / `:focus-within`. The canvas always renders at preview dimensions.

## PDF: PagedJS integration

### New file: `src/lib/export/renderPrintDocument.ts`

A fresh function that walks `data.blocks` and produces a complete HTML document tailored for paged printing. Direct render — not a transformation of `renderEmail` output. Returns a full `<!doctype html>...</html>` string.

Skeleton:

```ts
export function renderPrintDocument(data: ProjectData): string {
  const header = findHeader(data.blocks);
  const footer = findFooter(data.blocks);
  const middle = data.blocks.slice(1, -1);

  return `<!doctype html>
<html lang="en">
${renderPrintHead(data)}
<body>
  <div class="running-header">${renderHeaderForPrint(header, data)}</div>
  <div class="running-footer">${renderFooterForPrint(footer, data)}</div>
  <main>
    ${middle.map((b, i) => `<div class="print-block">${renderBlockForPrint(b, i, data)}</div>`).join('\n')}
  </main>
</body>
</html>`;
}
```

`renderHeaderForPrint` / `renderFooterForPrint` / `renderBlockForPrint` produce print-friendly HTML using the same `data` source. They may reuse formatting helpers (`htmlEscape`, `attrEscape`, `urlSafe`) but produce different markup than `renderEmail` — no MSO comments, no `width="710"` outer table, modern CSS-driven layout.

For each middle block type (`product-section`, `hero`, `article`, `cta-banner`), `renderBlockForPrint` switches on `block.type` and produces the appropriate HTML.

### Stylesheet (in `renderPrintHead`)

```css
@page {
  size: A4 portrait;
  margin: 32mm 12mm 32mm 12mm;
  @top-center    { content: element(header-region); }
  @bottom-center { content: element(footer-region); }
}

body { font-family: Arial, sans-serif; }
main { /* content area */ }

.running-header { position: running(header-region); }
.running-footer { position: running(footer-region); }

.print-block {
  break-inside: avoid;
  page-break-inside: avoid;  /* legacy alias for older renderers */
}

/* Hide the running elements at their original document location;
   PagedJS will pull them into @top-center / @bottom-center. */
.running-header, .running-footer { display: none; }
```

The `@page` margin (32mm top, 32mm bottom) is generous to leave room for header and footer content. Final margin values are chosen by measuring the rendered header/footer heights of the existing templates; if header heights are unusually tall (e.g., logo + banner image), the margin may need tuning per template — initial implementation uses a single global value, with a TODO for per-template margins if needed.

### Modified file: `src/lib/export/buildPrintHtml.ts`

Responsibility split:

- **`renderPrintDocument(data)`** produces a self-contained HTML document with the print stylesheet (`@page`, `position: running()`, `break-inside: avoid`) but **no** PagedJS `<script>` and **no** print toolbar. That keeps it a pure render function for snapshot testing.
- **`buildPrintHtml(data)`** is the thin top-level wrapper used by the `/print` route. It calls `renderPrintDocument(data)`, then injects (a) the `<script src="/vendor/paged.polyfill.js">` tag before `</body>`, (b) the print toolbar `<div class="no-print pagedjs_not_pageable">…</div>` as the first child of `<body>`, and (c) the auto-print script that hooks `PagedConfig.after`.

The current signature `buildPrintHtml(emailHtml: string): string` changes to `buildPrintHtml(data: ProjectData): string`. The single call site (`/print` route) is updated.
- The `window.print()` auto-trigger waits for the PagedJS `rendered` event rather than `window.load`:

```js
window.addEventListener('load', () => {
  if (window.PagedConfig) {
    window.PagedConfig.after = () => {
      requestAnimationFrame(() => requestAnimationFrame(() => window.print()));
    };
  } else {
    // Fallback if PagedJS failed to load
    requestAnimationFrame(() => requestAnimationFrame(() => window.print()));
  }
});
```

The current signature `buildPrintHtml(emailHtml: string): string` changes to `buildPrintHtml(data: ProjectData): string`. The single call site (`/print` route) is updated.

### Modified file: `src/app/w/[slug]/p/[id]/print/route.ts`

Changes from current:

- Removes `import { renderEmail }`.
- Calls `buildPrintHtml(data.data as ProjectData)` instead of `buildPrintHtml(renderEmail(data.data as ProjectData))`.
- Same auth, same caching headers, same not-found behavior.

### New vendored asset: `public/vendor/paged.polyfill.js`

The PagedJS polyfill bundle, vendored at a pinned version. Approximate size 150KB minified. Served as a static asset; no third-party network dependency at print time. The license is MIT, attribution kept in the file header.

The npm package `pagedjs` is added as a dev dependency for any build-time tooling, but the runtime asset is the standalone polyfill served as a static file (PagedJS publishes a UMD build for exactly this use case).

## Edit-fidelity audit

Every affordance listed below currently affects layout in edit mode. Fix pattern: take it out of normal flow.

### 3.1 Bullet drag-grip — `EditableBulletList.tsx`

**Current:** `<li>` contains `<button class="bullet-grip" min-w-[28px] min-h-[28px]>` immediately before `<EditableText>`. Inline-flex; grip eats 28px of horizontal space.

**Fix:**

- `<li>` becomes `position: relative`.
- `.bullet-grip` becomes `position: absolute; left: -24px; top: 50%; transform: translateY(-50%);` so it sits in the gutter.
- `.bullet-row:hover .bullet-grip`, `.bullet-row:focus-within .bullet-grip` set `opacity: 1`; default `opacity: 0`.
- The `<ul>` `padding-left` becomes `40px` in **both** edit and preview modes (currently `20px` in edit, `40px` in preview — second source of drift).
- The early-return `mode === 'preview'` branch in `EditableBulletList` stays as-is. Once the edit-mode CSS matches preview, both branches render visually identical output, but the preview branch produces simpler DOM (plain `<ul>` / `<li>` with no DndContext, no contenteditable, no sortable IDs). Keeping the split costs nothing and benefits a11y tools that may inspect role/aria attributes.

### 3.2 CTA URL pencil — `EditableLink.tsx` (used inside CTA `<a>`)

**Current:** Inline-flex `<button>` + 14px icon next to the CTA text inflates the button width.

**Fix:**

- The CTA `<a>` becomes `position: relative` (where it isn't already).
- `EditableLink`'s outer `<span>` becomes `position: absolute; right: -28px; top: 50%; transform: translateY(-50%);` relative to the CTA.
- Default `opacity: 0; pointer-events: none;`. The CTA's `:hover` / `:focus-within` selector reveals it: `opacity: 1; pointer-events: auto;`.
- The popover (URL input box that opens on click) keeps its current `absolute` positioning logic — no change there.
- The `inline-link-wrap inline-flex items-center gap-1` `<span>` wrapper around CTA-text + EditableLink can stay or be flattened; either works.

### 3.3 EditableImage hover overlay — `EditableImage.tsx`

**Current:** Likely renders an alt-text input or upload button alongside the image. Needs inspection during implementation to confirm exact mechanism.

**Fix (general pattern):**

- Image lives in a `position: relative` wrapper.
- Any edit controls (alt-text input, upload button, replace button) live in absolutely-positioned overlay elements (`position: absolute; top/left/right/bottom: 0` or similar).
- Overlays default to `opacity: 0; pointer-events: none;`, revealed on `:hover` / `:focus-within`.
- The image's bounding box exactly matches preview dimensions.

### 3.4 EditableText padding / focus styling — `EditableText.tsx`

**Current:** A `contenteditable` `<span>` or `<div>`. May have `padding`, `border`, or `outline` styles that add to its bounding box in edit mode.

**Fix:**

- Any `padding` / `border` / non-zero `outline-width` on `EditableText` in non-focused state is removed.
- A focus ring (`outline: 2px solid var(--brand); outline-offset: 2px;`) is added via `:focus-visible` — `outline` doesn't reserve space.
- The empty-placeholder rendering (when the field has no value) uses `::before { content: attr(data-placeholder); opacity: 0.5; }` rather than a separate inline element. Placeholder text occupies the same space as real text.

### 3.5 Block selection outline — `.section-wrap.selected` rule

**Current:** `outline: 2px solid var(--color-brand); outline-offset: -2px;`. `outline` doesn't affect layout — already correct.

**Action:** No change. Verify by inspection.

### 3.6 BlockToolbar reveal — `BlockToolbar.tsx`

**Current:** `position: absolute top-2 right-2 z-10`. Already out of flow.

**Action:** No change. Verify by inspection.

### 3.7 SectionInsertBar — `SectionInsertBar.tsx`

**Current:** Renders a `h-3 my-1` (12px tall) bar between every two blocks in edit mode. Occupies vertical space and pushes blocks apart vs preview.

**Fix:**

- The visible bar becomes `position: absolute; top: -6px; left: 0; right: 0; height: 12px;` inside a parent wrapper that is `position: relative; height: 0;` (zero-height container).
- A larger transparent hit region (`::before { position: absolute; top: -8px; height: 24px; ... }`) ensures the hover target is reachable even though the visible bar has zero flow height.
- Bar visibility: `.section-insert-bar:hover .section-insert-btn`, `.section-insert-btn:focus-visible` set `opacity: 1`.

### 3.8 Wrap-class normalization

All four canvas block views (`ProductSectionView`, `HeroBlockView`, `ArticleView`, `CTABannerView`) use `className="section-wrap <type>-wrap"`. The `.section-wrap` class drives the toolbar fade-in. No change needed here; the `block-wrap` rename is deferred per non-goals.

### 3.9 Edit-mode `<ul>` `paddingLeft` inconsistency

Already covered in §3.1 — both edit and preview render at `padding-left: 40px`.

## Print HTML structure: per-block

For reference, here is what each block type produces in print:

- **Header (`renderHeaderForPrint`):** A `<div class="print-header">` containing the logo image, title text, banner image, and section heading text. Uses modern CSS — no `<table width="710">` boilerplate. Layout: centered, max-width 710px, styled with the global font-family and background.

- **Footer (`renderFooterForPrint`):** A `<div class="print-footer">` containing the banner image, company name, address, contact links, and social icons. Same approach.

- **Product Section (`renderProductSectionForPrint`):** Two-column flex layout: image left, text right, alternating per `index % 2 === 1` (matches `renderEmail` zebra). Bullets as `<ul>`.

- **Hero, Article, CTA Banner:** Same shape as their canvas views but with simplified markup (no `useSortable`, no `EditableText` wrappers — just static HTML with the content values).

The HTML is built with the same `htmlEscape` / `attrEscape` / `urlSafe` helpers; XSS escaping behaves identically to the email render.

### Manual smoke checklist (documented in spec)

After the implementation lands, verify manually:

1. Open each of the 5 templates in the editor.
2. Click "Download HTML" — confirm it still works and produces the email HTML (no regression).
3. Open `/w/<slug>/p/<id>/print` — confirm the print view renders.
4. Press Cmd/Ctrl+P (or the toolbar button) — confirm:
   - Header is at the top of every page.
   - Footer is at the bottom of every page (including the last, even if the last page is sparse).
   - No product section / hero / article / cta-banner is split across two pages.
   - Page count is sensible for the template's content volume.
5. In edit mode at 100% zoom, eyeball the canvas vs preview mode at 100% zoom for one product, one hero, one article, one cta-banner block. Bullet spacing, CTA button widths, and block boundaries should be visually identical.

### No E2E test for the print route

Playwright doesn't easily verify printed PDF output cross-platform. The print HTML structure is unit-tested; visual fidelity is manually verified.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| PagedJS modifies DOM heavily and may conflict with the auto-print logic. | The auto-print waits for the PagedJS `rendered` event (via `PagedConfig.after`), not `window.load`. Falls back to direct `window.print()` if PagedJS isn't loaded. |
| `@page` margin (32mm top/bottom) may not be enough for tall headers (e.g., GlobalTT's logo + banner + section heading). | First implementation uses a global 32mm value. Manual smoke checks each of the 5 templates. If insufficient, a per-template or per-document override is added — flagged but not pre-implemented. |
| Vendored `paged.polyfill.js` size (~150KB) adds to the print route's payload. | The print route is rarely loaded (only when printing). 150KB is acceptable for that frequency. Vendored, not loaded on the main editor canvas. |
| Absolute-positioning the bullet-grip in the gutter may break on narrow viewports (left edge cut off). | The canvas is fixed-width (max 710px) and centered; the gutter region is part of the visible canvas, not off-screen. Verified by manual smoke. |
| `EditableImage`'s exact current behavior is not fully audited in the spec; the fix is stated generally. | Implementation task for `EditableImage` includes a "read the current component first" step before applying the fix pattern. Plan task will document the specific changes once the file is read. |
| Removing the early-return for `preview` mode in `EditableBulletList` may break a subtle accessibility behavior. | Implementation keeps the early-return if removing it doesn't simplify the code; the unified render path is an optimization, not a requirement. |
